# Життєвий цикл моделі та інтеграція з бекендом

Створення математичної моделі є лише частиною задачі. Для того щоб рекомендаційна система працювала швидко та не
блокувала роботу платформи **Smart-Study**, її глибоко інтегровано в асинхронну архітектуру Django з використанням
патерну Singleton, брокера фонових задач Celery та системи кешування Redis.

## 1. Подієво-орієнтоване тренування (Event-Driven Training)

Модель не перетреновується при кожному запиті або за жорстким розкладом. Натомість реалізовано подієво-орієнтований
підхід (Event-Driven Architecture) за допомогою **Django Signals**.

1. **Відстеження подій:** Спеціальний слухач (`@receiver(post_save, sender=Course)`) моніторить зміни у моделі курсів.
2. **Фільтрація:** Перетренування запускається виключно тоді, коли створюється **новий опублікований курс** (
   `created and instance.is_published`).
3. **Асинхронне виконання:** Розрахунок TF-IDF та матриці косинусної схожості може займати певний час. Щоб викладач,
   який публікує курс, не чекав завершення цих розрахунків, викликається фонова задача Celery —
   `train_course_recommendations.delay()`.
4. **Скидання кешу:** Після успішного оновлення моделі (`courses_recommender.train()`), Celery-воркер автоматично очищує
   старий кеш рекомендацій (`invalidate_all_course_recommendations_cache`), щоб студенти одразу почали отримувати
   актуальні дані.

---

## 2. In-Memory Singleton (Ліниве завантаження)

Файлова система не пристосована для постійних операцій читання під навантаженням. Тому об'єкт `CourseRecommender`
реалізований за патерном **Singleton** (`courses_recommender = CourseRecommender()`).

Коли клієнт робить запит:

1. Система перевіряє, чи завантажена матриця схожості (`.pkl` файл) в оперативну пам'ять (RAM) сервера.
2. Якщо ні (наприклад, після перезапуску сервера), відбувається **ліниве завантаження (Lazy Loading)**.
3. Усі наступні запити використовують готову матрицю з RAM, що зменшує час пошуку рекомендацій до $O(1)$.

---

## 3. Кешування результатів (Redis Caching Layer)

Навіть маючи швидку ML-модель у пам'яті, процес перетворення списку ідентифікаторів (IDs) у повноцінний JSON-відповідь
вимагає звернення до PostgreSQL (для отримання метаданих курсу та інформації про авторів). Для оптимізації цього
вузького місця використовується багаторівневе кешування:

- **Унікальні ключі:** Результат для кожного курсу кешується під унікальним ключем, що враховує статус проходження (
  наприклад, `recommendations:{course_id}-{status}`).
- **TTL (Час життя):** За замовчуванням рекомендації живуть у Redis 24 години (`CACHE_TIMEOUT = 60 * 60 * 24`), якщо їх
  не інвалідує система при публікації нового контенту.
- **Зменшення навантаження на БД:** Якщо рекомендації знайдені в кеші, бекенд взагалі не звертається до бази даних і
  повертає відповідь за лічені мілісекунди.

---

## 4. Оптимізація ORM та збереження порядку

Коли система вперше генерує рекомендації (до кешування), ML-модель повертає список ID відсортований за релевантністю (
наприклад, `[45, 12, 89]`). Проте стандартний запит `Course.objects.filter(id__in=...)` порушує цей порядок, повертаючи
курси відсортовані за їхнім ID у базі.

Щоб зберегти правильний порядок релевантності, у сервісі `get_course_recommendations` застосовано оптимізацію за
допомогою SQL-оператора `CASE`:

```python
preserved_order = Case(*[When(pk=pk, then=pos) for pos, pk in enumerate(recommended_ids)])
courses_qs = Course.objects.filter(id__in=recommended_ids).order_by(preserved_order)
```

Це дозволяє відсортувати об'єкти безпосередньо на рівні бази даних (PostgreSQL), уникаючи дорогоцінного сортування в
Python.

---

## 5. Інтеграція з REST API клієнта

Кінцевий зв'язок із клієнтським додатком реалізований через асинхронний REST-ендпоінт `CourseRecommendationsView`.

**Приклад взаємодії:**

- Клієнт відправляє GET-запит: `GET /api/courses/course-recommendations/{course_id}/?status=passed&limit=6`
- **Валідація:** Ендпоінт суворо перевіряє параметри (статус має бути лише `passed` або `failed`, а ліміт — цілим числом
  до 100).
- **Виконання:** Викликається кеш-сервіс.
- **Результат:** Клієнт отримує JSON із детальною інформацією про курси, готову для відображення в інтерфейсі (картки
  курсів, автори, рівень складності).

Завдяки використанню `async def get` та методів `sync_to_async`, цей ендпоінт не блокує цикл подій (Event Loop)
сервера (ASGI), що дозволяє обслуговувати тисячі одночасних підключень студентів з мінімальною затримкою.
