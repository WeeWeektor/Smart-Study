# Контейнеризація та оркестрація (Docker Architecture)

Інфраструктура платформи **Smart-Study** повністю контейнеризована за допомогою Docker та керується через
`docker-compose`. Проєкт складається з 8 взаємопов'язаних мікросервісів, які працюють у єдиній ізольованій мережі
`smartStudy_network`.

## 1. Топологія контейнерів

- **`nginx` (Web Server / Reverse Proxy):** Єдина точка входу в систему. Обробляє SSL-сертифікати (Let's Encrypt),
  редиректить HTTP на HTTPS, маршрутизує API-запити до бекенду, а також самостійно та швидко віддає статику, медіа та
  скомпільований фронтенд.
- **`backend` (Django ASGI):** Основний сервер бізнес-логіки. Працює на базі `gunicorn` з воркерами `UvicornWorker` для
  підтримки асинхронності високонавантажених ендпоінтів.
- **`frontend_builder` (React/Vite):** Службовий контейнер з Multi-stage збіркою. Компілює React-код у статику (
  HTML/JS/CSS) і копіює його у спільний volume `frontend_dist`, звідки його читає Nginx.
- **`postgres` (Relational DB):** Сховище для користувачів, транзакцій та зв'язків. Має налаштований `healthcheck` для
  гарантії того, що бекенд почне запуск лише після повної готовності БД.
- **`mongodb` (NoSQL DB):** Сховище для гнучких ієрархічних структур курсів та тестів.
- **`redis` (Cache & Message Broker):** Використовується як in-memory кеш для сповіщень та профілів, а також як брокер
  повідомлень для Celery.
- **`celery_worker` & `celery_beat`:** Контейнери для виконання асинхронних фонових задач (відправка email, масові
  розсилки) та виконання задач за розкладом (ML тренування, нічні бекапи).

## 2. Мережева безпека та ізоляція

Система побудована за принципом "найменших привілеїв" (Zero Trust на рівні мережі):

- Назовні (в інтернет) відкриті **лише порти 80 та 443** через контейнер `nginx`.
- Контейнери бекенду (`backend`, `celery`) та баз даних (`postgres`, `mongodb`, `redis`) "спілкуються" між собою
  виключно через внутрішню Docker-мережу за внутрішніми DNS-іменами. Доступ до них ззовні напряму неможливий (крім
  прокинутих портів для адміністрування, які захищені фаєрволом Azure).

## 3. Управління даними (Volumes Sharing)

Для ефективної роботи системи без дублювання файлів використовується система спільних томів (Shared Volumes):

- **`frontend_dist`:** `frontend_builder` компілює туди код $\rightarrow$ `nginx` читає його (`ro` - read-only) та
  віддає клієнтам як статичний сайт.
- **`staticfiles` та `media`:** Django (`backend`) збирає туди файли адмінки та завантаження користувачів $\rightarrow$
  `nginx` читає їх безпосередньо, обходячи Python-сервер, що критично збільшує швидкість завантаження контенту.
- **`postgres_data` та `mongodb_data`:** Забезпечують персистентність (збереження) даних баз при перезапуску або
  оновленні контейнерів.

## 4. Конфігурація Nginx

Конфігурація `nginx/conf.d` оптимізована для сучасних веб-додатків:

1. **SPA Routing:** Запити на корінь `/` перенаправляються на `index.html` фронтенду, дозволяючи React Router керувати
   навігацією (`try_files $uri $uri/ /index.html`).
2. **API Proxy:** Запити, що починаються з `/api/` або `/admin/`, проксіюються на upstream `asgi_backend` з передачею
   реальних IP-адрес користувачів через `X-Forwarded-For`.
3. **Upload Limits:** Встановлено `client_max_body_size 500M` для можливості завантаження відеоматеріалів лекцій.
