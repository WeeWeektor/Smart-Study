# Архітектура клієнтської частини (Feature-Sliced Design)

У проєкті **Smart-Study** фронтенд розроблено з використанням архітектурної методології **Feature-Sliced Design (FSD)**.
Це дозволяє структурувати код таким чином, щоб він був легким для тестування, розширення та підтримки.

## 1. Загальна концепція шарів (Layers)

Код у директорії `frontend/src/` розділений на ієрархічні шари. Головне архітектурне правило: **модуль може імпортувати
лише те, що знаходиться на шарах нижче.**

| Шар (Layer)  | Опис                                                                        | Приклади в проєкті                                 |
| :----------- | :-------------------------------------------------------------------------- | :------------------------------------------------- |
| **App**      | Ініціалізація додатку, глобальні стилі та провайдери.                       | `App.tsx`, `main.tsx`, `index.css`.                |
| **Pages**    | Композиційні блоки, що формують цілі екрани додатку.                        | `Calendar.tsx`, `CourseReview.tsx`, `Profile.tsx`. |
| **Widgets**  | Складні самодостатні блоки інтерфейсу, що складаються з декількох Features. | `CourseSidebar`, `LearningStats`, `ProfileHeader`. |
| **Features** | Користувацькі дії, що приносять бізнес-цінність (форми, кнопки).            | `login`, `register`, `upload-avatar`.              |
| **Entities** | Бізнес-сутності та дані, якими оперує додаток.                              | `profile`, `review`, `notification`.               |
| **Shared**   | Перевикористовувані ресурси, UI-kit та інфраструктурний код.                | `ui/button`, `api-client`, `lib/i18n`.             |

---

## 2. Структура директорій проєкту (Спрощена)

Нижче наведено **часткову (базову)** структуру директорій `frontend/src/`. Вона демонструє лише ключові архітектурні
вузли згідно з методологією FSD. Реальне дерево проєкту значно ширше, проте ця схема ілюструє головний принцип
розподілу:

```text
frontend/src/
├── app/                        # Налаштування застосунку, провайдери, роутер
│   ├── App.tsx
│   ├── index.css
│   └── ...                     # Інші файли ініціалізації
│
├── pages/                      # Сторінки (маршрути)
│   ├── calendar/
│   ├── course-review/
│   ├── Profile.tsx
│   └── ...                     # Інші екрани додатку
│
├── widgets/                    # Складні складені компоненти
│   ├── auth/                   # Віджети екранів авторизації
│   ├── course/                 # Шапка та сайдбар курсу
│   ├── profile/                # Статистика та вкладки профілю
│   └── ...
│
├── features/                   # Бізнес-сценарії та дії користувача
│   ├── auth/                   # Логіка авторизації
│   ├── course/                 # Дії з курсами (wishlist, enrollment)
│   ├── lesson-type-fields/     # Динамічні форми для контенту
│   └── ...
│
├── entities/                   # Бізнес-сутності (моделі та API)
│   ├── notification/           # Структура сповіщень
│   ├── profile/                # Дані користувача та досягнення
│   ├── review/                 # Відгуки до курсів
│   └── ...
│
└── shared/                     # Спільний код (найнижчий рівень)
    ├── api/                    # Налаштування Axios, Interceptors
    ├── hooks/                  # Загальні хуки (useTheme, useLanguage)
    ├── lib/                    # Утиліти (i18n, форматування дат)
    ├── ui/                     # Базові UI компоненти (Button, Input, Modal)
    └── ...                     # Інший перевикористовуваний код
```

## 3. Деталізація ключових рівнів

### 3.1. Shared (Спільний шар)

Найнижчий рівень, який не містить жодної специфічної бізнес-логіки системи LMS. Його модулі можна безболісно перенести в
будь-який інший проєкт.

- **`shared/ui`**: Власна бібліотека компонентів (Button, Input, Avatar, Card, Modal), стилізована за допомогою Tailwind
  CSS.
- **`shared/api`**: Налаштований клієнт Axios та перехоплювачі (interceptors) для безпечної роботи з сесіями та
  обов'язкової передачі CSRF-токенів.
- **`shared/lib`**: Загальні утиліти для форматування дат, роботи з Cookies та конфігурація інтернаціоналізації (
  перемикання мов `uk` / `en` через i18n).

### 3.2. Entities (Бізнес-сутності)

Описують предметну область: "про що" цей проєкт. Вони містять типи TypeScript, базові запити до API та
стейт-менеджмент (наприклад, React Query) для конкретної сутності.

- **`profile`**: Зберігає дані про користувача, його прогрес, статистику навчання та здобуті нагороди (achievements).
- **`review`**: Моделі даних відгуків про курси та утиліти для підрахунку загального рейтингу.

### 3.3. Features (Можливості)

Реалізують конкретні сценарії взаємодії користувача з системою. Features можуть використовувати Entities та Shared, але
ніколи не імпортують інші Features.

- **`lesson-type-fields`**: Складна логіка динамічного рендерингу та управління формами залежно від типу уроку (Video,
  Text, Code, Presentation).
- **`course-recommendation`**: Логіка запиту та обробки персоналізованих рекомендацій, згенерованих ML-ядром платформи.

## 4. Архітектурні правила та обмеження

Для підтримки чистоти коду та уникнення "спагетті-коду" в проєкті суворо дотримуються наступні правила:

1. **Односпрямованість залежностей (Unidirectional Dependencies):** Імпорти дозволені лише зверху вниз. Наприклад, шар
   `Pages` може імпортувати `Widgets` та `Features`, але `Features` **не може** імпортувати `Widgets` чи `Pages`.
2. **Публічний API (Public Interfaces):** Кожна директорія (slice) має містити файл `index.ts`, який є єдиною точкою
   входу. Заборонені "глибокі імпорти" зсередини модуля (наприклад,
   `import { Button } from 'shared/ui/button/button.tsx'`). Всі зовнішні модулі мають звертатися лише до експортів з
   `index.ts`.
3. **Внутрішня сегментація:** Кожен модуль (наприклад, у `features/auth`) розділений за функціональним призначенням на
   стандартні сегменти:
   - `ui/` — React-компоненти.
   - `model/` — бізнес-логіка, стор, хуки, типи.
   - `api/` — специфічні запити до бекенду.
   - `lib/` — допоміжні функції модуля.

---

## 5. Переваги обраної архітектури для Smart-Study

1. **Передбачуваність та Масштабованість:** Додавання нових типів завдань (наприклад, аудіо-уроків) або нових алгоритмів
   ML-рекомендацій не впливає на стабільність існуючих модулів.
2. **Високий рівень ізоляції:** Зміна логіки розрахунку статистики в профілі зачіпає лише відповідний віджет та
   сутність, залишаючи решту додатку (наприклад, плеєр курсів) недоторканою. Це мінімізує ризик регресійних багів.
3. **Надійність (Типізація):** Чіткий розподіл сутностей дозволяє максимально ефективно використовувати можливості
   TypeScript, запобігаючи виникненню помилок розбіжності даних між фронтендом та API ще на етапі компіляції.
