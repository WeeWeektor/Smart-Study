# Архітектура бекенду та сервісний шар

Бекенд-частина платформи **Smart-Study** розроблена на базі фреймворку **Django** з використанням **Django REST
Framework (DRF)**. Архітектура побудована навколо ідеї декомпозиції бізнес-логіки у виділений шар сервісів (Service
Layer).

Цей підхід гарантує чистоту коду (Clean Code), спрощує тестування та запобігає появі антипатернів "Fat Models" (
перевантажені моделі) чи "Fat Views" (перевантажені контролери).

---

## 1. Логіка розподілу на застосунки (Django Apps)

Система розділена на декілька функціональних модулів (додатків), кожен з яких відповідає за свою ізольовану доменну
область:

| Додаток              | Зона відповідальності                                                                                                            |
| :------------------- | :------------------------------------------------------------------------------------------------------------------------------- |
| **`users`**          | Управління профілями, кастомна модель користувача, автентифікація (включно з OAuth через Google/Facebook) та статистика профілю. |
| **`courses`**        | Основне ядро: управління курсами, модулями, уроками різних типів та тестами. Відстеження прогресу та видача сертифікатів.        |
| **`notifications`**  | Внутрішня система сповіщень для користувачів та викладачів. Тісно інтегрована з Celery для асинхронної обробки подій.            |
| **`users_calendar`** | Планування навчального процесу, створення персональних подій та розрахунок графіку завершення курсів з нагадуваннями.            |
| **`common`**         | Спільні утиліти, middleware, базові класи репозиторіїв та декоратори, що використовуються іншими додатками проєкту.              |
| **`core`**           | Базові налаштування проєкту.                                                                                                     |
| **`ml_model`**       | Ізольований модуль рекомендаційної системи на базі алгоритмів машинного навчання (TF-IDF, Cosine Similarity).                    |

## 2. Патерн Service Layer (Сервісний шар)

У проєкті Smart-Study застосовано суворий архітектурний підхід: уся бізнес-логіка винесена з `views.py` та `models.py` у
спеціальні модулі, що знаходяться в директоріях `services/` кожного додатку.

### Чому обрано цей підхід:

1. **DRY (Don't Repeat Yourself):** Один і той самий сервіс може використовуватися в API-представленнях,
   адміністративній панелі або фонових задачах Celery без дублювання коду.
2. **Ізольоване тестування:** Сервіси — це незалежні Python-класи або функції, які легко покривати Unit-тестами
   ізольовано від HTTP-запитів (Request/Response об'єктів).
3. **Читабельність:** Файли `views.py` відповідають виключно за обробку HTTP-запиту, валідацію вхідних даних (через
   Serializers), виклик потрібного сервісу та форматування HTTP-відповіді.

### Приклади реалізації:

- **`courses/services/course_actions_service/`**: містить окремі функції/класи для створення, оновлення, публікації та
  видалення курсів.
- **`courses/services/test_actions_service/`**: містить складну бізнес-логіку проходження тестів, перевірку відповідей
  та інвалідацію кешу.
- **`users/services/profile_update_service.py`**: обробляє логіку зміни даних профілю, включаючи безпечне завантаження
  медіафайлів.

## 3. Гібридна логіка роботи з даними (Polyglot Persistence)

Особливістю архітектури є одночасне використання різних типів сховищ, доступ до яких інкапсульовано через сервісний шар:

- **Relational Data:** Використання Django ORM для роботи з **PostgreSQL** (збереження транзакційних даних: користувачі,
  фінанси, базові метадані курсів, статуси запису).
- **Document Data:** Спеціалізований сервіс `common/services/mongo_repository.py` забезпечує інтерфейс для безпечної
  роботи з **MongoDB**, де зберігаються динамічні, гнучкі JSON-структури (контент уроків, питання та результати тестів).
- **File Assets:** Сервіси завантаження (наприклад, `supabase.py` та `picture_service.py`) відповідають за пряму
  взаємодію з хмарним об'єктним сховищем **Supabase**, приховуючи від бізнес-логіки деталі роботи з API сховища.

## 4. Асинхронність та кешування

Для забезпечення високої швидкодії та стійкості до навантажень у сервісах активно використовуються оптимізаційні
патерни:

### Cache Service

Кожен великий додаток має свою директорію `services/cache_service/`. Кешування (через **Redis**) дозволяє:

- Кардинально зменшити кількість запитів до бази даних для операцій читання (списки курсів, статуси підписок).
- Зберігати проміжні стани системи (наприклад, стан сповіщень у `notifications_cache.py`).

### Background Tasks (Celery)

Сервіси щільно інтегровані з файлами `tasks.py`. Задачі, що вимагають багато часу або залежать від сторонніх API, не
виконуються в основному Request-Response циклі.
Наприклад, коли користувач успішно завершує курс, сервіс делегує задачу генерації PDF-сертифіката або перерахунку
ML-матриці рекомендацій асинхронному **Celery Worker-у**.

## 5. Детальна структура директорій (на прикладі `courses`)

Нижче наведено структуру сервісного шару модуля `courses`, що демонструє рівень декомпозиції бізнес-логіки:

```text
backend/courses/services/
├── builder_json/                 # Патерн Builder для збірки складних JSON-відповідей
├── cache_service/                # Логіка інвалідації та запису в Redis-кеш
├── certificate_actions_service/  # Генерація та видача PDF/PNG сертифікатів
├── course_actions_service/       # CRUD операції над курсами та обчислення статистики
├── course_by_user/               # Операції користувача (додавання у Wishlist, Enrollment)
├── lesson_actions_service/       # Робота з уроками та конвертація Markdown-контенту
├── structure_course_module...    # Управління ієрархією структури курсу (модулі, уроки, тести)
└── test_actions_service/         # Валідація спроб проходження тестів
```

## 6. Висновки по архітектурі бекенду

Використання чітко ізольованих Django-додатків та винесення логіки в сервісний шар дозволило створити систему, яка:

- **Має високу підтримуваність (Maintainability):** Зміна логіки обчислення балів тесту зачіпає лише один файл у
  `test_actions_service`, не вимагаючи втручання в контролери (API-ендпоінти) або моделі.
- **Є відмовостійкою:** Помилка при генерації сертифіката чи відправці імейла (у Celery-тасці) не блокує користувачеві
  доступ до платформи, оскільки процес виконується асинхронно у фоні.
- **Відповідає індустріальним стандартам (Enterprise Standards):** Така декомпозиція є класичним прикладом професійної
  архітектури в складних продуктових розробках. Орієнтація на інженерний підхід дозволяє створювати чистий код, який
  легко розширювати та презентувати.
