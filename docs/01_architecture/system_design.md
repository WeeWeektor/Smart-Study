# Загальна архітектура системи (System Design)

Платформа **Smart-Study** побудована за архітектурним патерном Client-Server із використанням слабкозв'язаних компонентів (Decoupled Architecture). Вона складається з незалежного клієнтського SPA-додатка, RESTful API бекенду та асинхронної підсистеми обробки фонових задач.

Такий підхід забезпечує високу масштабованість, незалежне розгортання модулів та ізоляцію відмов.

## 1. Високорівнева схема компонентів

Система розділена на п'ять основних логічних рівнів:

1. **Presentation Layer (Клієнт):** React SPA (Single Page Application).
2. **Gateway / Proxy Layer:** Nginx (Reverse Proxy, SSL Termination).
3. **Application Layer (Бекенд):** Django, Django REST Framework.
4. **Asynchronous Processing Layer:** Celery Workers + Celery Beat.
5. **Data Storage Layer:** Поліглотне сховище (PostgreSQL, MongoDB, Redis, Supabase).

---

## 2. Взаємодія компонентів (Data Flow)

Усі зовнішні запити до системи проходять через єдину точку входу — веб-сервер **Nginx**, який виконує роль зворотного проксі-сервера.

### Життєвий цикл типового HTTP-запиту:

1. **Клієнт** (браузер) ініціює HTTPS запит до `smart-study.me`. Під час авторизації браузер автоматично прикріплює `httpOnly` сесійну куку. Для мутуючих запитів (POST, PUT, DELETE) фронтенд додатково передає **CSRF-токен** у заголовках.
2. **Nginx** приймає запит, виконує SSL-термінацію (дешифрування трафіку) та маршрутизує його:
   - Якщо це запит до статичних файлів фронтенду — віддає скомпільовані файли React-додатка.
   - Якщо шлях починається з `/api/` або `/admin/` — проксіює запит до контейнера **Django** (через WSGI/ASGI сервер, Gunicorn/Uvicorn).
3. **Django** обробляє запит:
   - **Authentication:** Ідентифікує користувача за допомогою Session ID, отриманого з куки (Session Authentication).
   - **Security Check:** Перевіряє наявність та валідність CSRF-токена для захисту від підробки міжсайтових запитів.
   - Звертається до **PostgreSQL** для перевірки прав доступу чи отримання реляційних даних.
   - Звертається до **MongoDB** для витягування складноструктурованих даних (наприклад, JSON-структури курсу чи питань тестів).
4. Якщо запит потребує важких обчислень (наприклад, перерахунок ML-рекомендацій), Django створює задачу і кладе її в чергу **Redis**.
5. Django повертає клієнту швидку відповідь (HTTP 200/201/202).

## 3. Опис ключових вузлів системи

### 3.1. Nginx (Reverse Proxy)

Використовується для забезпечення безпеки та балансування навантаження на рівні інфраструктури.

- **Відповідальність:** Маршрутизація трафіку, блокування шкідливих запитів, управління CORS політиками на рівні сервера, підтримка HTTPS (сертифікати Let's Encrypt).

### 3.2. Django + DRF (Application API)

Основне ядро бізнес-логіки. Реалізовано за принципом "Товсті моделі / сервіси, тонкі контролери (views)".

- **Відповідальність:** Багаторівнева безпека (Session Auth, валідація CSRF, захист від XSS), CRUD операції, управління транзакціями, валідація даних, авторизація користувачів (у тому числі OAuth). Замість розміщення бізнес-логіки безпосередньо у Views, використовується сервісний шар (директорії `services/` всередині додатків), що спрощує юніт-тестування та ізолює логіку.

### 3.3. Celery та Redis (Asynchronous Tasks)

Синхронна обробка важких задач у рамках HTTP-запиту призводить до блокування потоків веб-сервера (Timeout). Тому архітектура передбачає винесення ресурсомістких процесів у фон.

- **Redis** виконує роль брокера повідомлень (Message Broker) та швидкого in-memory кешу.
- **Celery Worker** — фоновий процес, який забирає задачі з Redis і виконує їх.
- **Celery Beat** — планувальник задач (Scheduler), який запускає періодичні процеси.

**Типові фонові задачі в системі:**

- Формування векторів TF-IDF та перерахунок матриці косинусної схожості для ML-рекомендацій.
- Генерація PDF/PNG сертифікатів після успішного проходження курсу.
- Розсилка email-сповіщень користувачам.
- Очищення застарілих сесій у базі даних.

### 3.4. Supabase (Blob Storage)

Для зберігання користувацького контенту (аватарки, обкладинки курсів, вкладені файли уроків) використовується зовнішнє хмарне об'єктне сховище (Object Storage), сумісне з S3 API.

- **Відповідальність:** Розвантаження основних баз даних та файлової системи сервера від зберігання великих бінарних об'єктів. Забезпечує швидку віддачу медіаконтенту через вбудований CDN.

## 4. Обґрунтування інженерних рішень

1. **Decoupled Client-Server:** Розділення React-фронтенду та Django-бекенду дозволяє масштабувати їх незалежно. Якщо навантаження на обчислення рекомендацій зросте, можна збільшити кількість реплік Django/Celery, не зачіпаючи сервер статики фронтенду.
2. **Безпека на базі Session Auth + CSRF:** Відмова від JWT на користь класичних сесій дозволила підвищити безпеку. Сесійні токени зберігаються в `httpOnly` куках, що унеможливлює їх викрадення через XSS-атаки. А обов'язкова перевірка CSRF-токенів надійно захищає API від міжсайтових підробок.
3. **Використання Redis як універсального інструменту:** Замість розгортання окремого RabbitMQ для черг і Memcached для кешування, обрано Redis, який ефективно виконує обидві ролі, що зменшує навантаження на інфраструктуру (Resource overhead) та спрощує конфігурацію Docker Compose.
4. **Винесення ML-модуля:** Інтеграція `scikit-learn` та моделей машинного навчання безпосередньо у Django-проєкт реалізована через відокремлену директорію `ml_model/`. Самі обчислення жорстко прив'язані до воркерів Celery, що гарантує відсутність блокування основного Event Loop сервера під час матричних обчислень.
