# Механізм авторизації та автентифікації

Підсистема безпеки **Smart-Study** побудована навколо класичної моделі сесій (Session-based Authentication) з
використанням асинхронних обробників (ASGI) для підвищення пропускної здатності.

## 1. Архітектурна стратегія безпеки

- **Сесії замість JWT:** Після успішної перевірки облікових даних сервер генерує унікальний ідентифікатор сесії (
  `sessionid`), який зберігається у браузері клієнта як `HttpOnly` cookie. Це унеможливлює крадіжку токена через
  XSS-атаки (JavaScript не має доступу до cookie).
- **CSRF-захист:** Оскільки використовуються сесії, система суворо вимагає наявність `csrftoken` для всіх мутуючих
  запитів (`POST`, `PATCH`, `DELETE`). Фронтенд отримує токен через спеціальний ендпоінт `/api/auth/get-csrf-token/`.
- **Ізоляція БД:** Створення користувача, його налаштувань (`UserSettings`) та профілю (`UserProfile`) обгорнуто в єдину
  атомарну транзакцію бази даних (`transaction.atomic`). Це гарантує, що у разі системної помилки в базі не залишаться "
  напівстворені" профілі.

---

## 2. Життєвий цикл реєстрації (Email Verification)

Система не дозволяє непідтвердженим користувачам користуватися платформою:

1. **Реєстрація:** Користувач вводить дані (ім'я, email, пароль, роль). Пароль хешується, а запис створюється зі
   статусом `is_active=False` та `is_verified_email=False`.
2. **Криптографічний підпис:** Замість зберігання тимчасових кодів у базі даних, бекенд використовує вбудований модуль
   `django.core.signing`. Створюється захищений токен, який містить зашифрований email користувача, і відправляється на
   пошту.
3. **Підтвердження:** При переході за посиланням сервер розшифровує токен. Токен має вбудований термін придатності (24
   години — `max_age=60 * 60 * 24`). Якщо токен валідний, акаунт активується.

---

## 3. Соціальна авторизація (Social OAuth2)

Для забезпечення безшовного входу (Single Sign-On) реалізовано інтеграцію з Google та Facebook через ендпоінти
`GoogleAuthView` та `FacebookAuthView`.

**Flow (потік виконання):**

1. Клієнтський додаток (React) відкриває вікно провайдера (Google/Facebook) і отримує від нього JWT-ідентифікатор (
   `credential`).
2. Цей токен передається на бекенд.
3. Сервіс `handle_oauth_login` валідує токен через API провайдера.
4. Якщо email вже існує, користувач автоматично авторизується. Якщо ні — бекенд "на льоту" створює новий акаунт,
   заповнюючи ім'я та пошту.
   _Конфіденційність:_ Платформа ніколи не отримує і не зберігає паролі користувачів від соціальних мереж.

---

## 4. Управління паролями та захист від спаму

Процес скидання пароля (Forgot Password) також використовує криптографічні токени з терміном дії 24 години.

**Захист від перенавантаження (Rate Limiting):**
Щоб зловмисники не могли використовувати ендпоінт для email-бомбінгу, у методі `ForgotPasswordView` реалізовано
блокування через Redis. Після запиту на скидання пароля, в кеш записується ключ `forgot_reset_{hash(email)}` на 5
хвилин. Будь-які повторні запити протягом цього часу відхиляються з HTTP-кодом `429 Too Many Requests`.

---

## 5. Кешування профілів (Performance Optimization)

Оскільки перевірка статусу користувача відбувається при кожному захищеному запиті, звернення до PostgreSQL мінімізовані.
Після успішного логіну (через email або OAuth), викликається метод `warm_user_cache(user)`. Він попередньо завантажує
всі дані профілю, налаштування та дозволені ролі в оперативну пам'ять (Redis), забезпечуючи миттєву відповідь сервера (
`O(1)`) для наступних запитів клієнта.
