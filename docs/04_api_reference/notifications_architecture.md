# Архітектура підсистеми сповіщень (Notifications System)

Підсистема сповіщень **Smart-Study** спроєктована для роботи в умовах високих навантажень. Оскільки освітня платформа
передбачає часті масові розсилки (наприклад, викладач відправляє анонс тисячам студентів), архітектура побудована на
принципах асинхронності, глибокого кешування та фонової обробки.

## 1. Стратегія отримання даних (Pull-модель)

Замість утримання тисяч постійних WebSocket-з'єднань, які споживають багато оперативної пам'яті сервера, обрано
оптимізовану **Pull-модель** (polling).

- **Механізм:** Клієнтський додаток самостійно опитує ендпоінт `/api/notifications/get_notifications/` (наприклад, при
  завантаженні сторінки або періодично).
- **Продуктивність:** Завдяки асинхронним обробникам (ASGI) та Redis, сервер здатен обробляти тисячі таких запитів на
  секунду без блокування основного потоку виконання.

## 2. Кешування та інвалідація (Redis)

Для мінімізації навантаження на PostgreSQL впроваджено агресивне кешування на рівні сервісів (`NotificationsCache`,
`CourseOwnerNotificationCache`).

- **Читання:** Запити користувачів на отримання сповіщень не досягають бази даних — вони миттєво віддаються з
  оперативної пам'яті (Redis).
- **Інвалідація:** Стан кешу є консистентним. Коли користувач викликає метод `mark_as_read` (прочитує одне або всі
  сповіщення), сервіс `MarkAsRead` атомарно оновлює БД (`is_read=True`) і миттєво інвалідує кеш (
  `invalidate_notifications_cache()`). Наступний Pull-запит завантажить свіжі дані.

## 3. Масові розсилки (Bulk Operations)

Відправлення анонсів студентам курсу (`CourseOwnerNotifications`) оптимізовано на рівні ORM.

- Замість створення записів у циклі (`O(N)` запитів до БД), система використовує `Notification.objects.bulk_create()`.
  Це дозволяє створити тисячі сповіщень за один `INSERT` запит до бази даних.
- Одразу після створення записів, система виконує групову інвалідацію кешу для всіх підписаних студентів (
  `invalidate_for_users_cache`).

## 4. Фонові задачі та Email-сповіщення (Celery)

Доставка email-сповіщень повністю відокремлена від HTTP-запитів клієнта, щоб уникнути таймаутів.

- **Celery Task:** Функція `send_bulk_notification_email_task.delay()` ставить розсилку в чергу фонових задач (брокер —
  Redis).
- **Чанкінг (Chunking):** Для запобігання блокування SMTP-сервера (наприклад, Gmail), масив адрес отримувачів
  розбивається на пакети по 50 адрес (`CHUNK_SIZE = 50`). У межах одного відкритого SMTP-з'єднання (`get_connection()`)
  відправляється цілий пакет листів, що критично зменшує час виконання розсилки.

## 5. Життєвий цикл та сегментація

Сповіщення сегментуються на "Активні" та "Архівовані" на рівні бекенду:

- **Активні:** Непрочитані сповіщення, або прочитані, але створені менше ніж 7 днів тому (`sent_at__gte=threshold`).
- **Архів:** Сповіщення, які були прочитані понад 7 днів тому.
  БД автоматично лімітує вибірку до 100 записів (`queryset[:100]`), гарантуючи стабільний час відповіді (Response Time)
  незалежно від того, як довго користувач зареєстрований на платформі.
