# Організація In-Memory кешування та хмарного сховища

Для забезпечення високої швидкодії, відмовостійкості та розвантаження основних баз даних (PostgreSQL та MongoDB),
архітектура **Smart-Study** використовує додаткові інструменти: **Redis** для тимчасового зберігання даних і черг задач,
та **Supabase Storage** для управління файлами.

## 1. In-Memory кешування та черги (Redis)

Redis у проєкті виконує дві ключові архітектурні ролі: проміжне сховище швидкого доступу (Cache) та брокер повідомлень (
Message Broker) для фонових задач.

### 1.1. Application Cache (Кешування даних)

Для зменшення навантаження на бази даних і пришвидшення віддачі контенту клієнту, результати важких запитів зберігаються
в оперативній пам'яті (RAM) сервера за допомогою Redis.

- **Що кешується:** Опубліковані курси (метадані + структура), поточний прогрес студентів та інші ресурсоємні вибірки.
- **Стратегія інвалідації (TTL):** Час життя кешу (Time-To-Live) налаштовується динамічно залежно від типу даних і
  становить від 10 хвилин до 1 години. При оновленні курсу автором кеш примусово інвалідується (скидається), щоб
  студенти отримували актуальну версію.

### 1.2. Message Broker для Celery (Фонові задачі)

Платформа генерує задачі, які неможливо виконати синхронно під час HTTP-запиту користувача (наприклад, масові розсилки
або обчислення ML-рекомендацій). Redis виступає надійним брокером, який приймає ці задачі та передає їх воркерам Celery.

- **Приклад реалізації (Масові сповіщення):**
  У системі реалізована асинхронна задача `send_bulk_notification_email_task`. Для уникнення блокування SMTP-сервера та
  тайм-аутів, відправка листів (HTML + Plain Text) розбивається на "чанки" (chunks) по 50 адрес. Celery-воркер поступово
  обробляє ці пакети у фоні, тоді як користувач (або адміністратор) миттєво отримує відповідь від сервера, що розсилку
  запущено.

## 2. Хмарне файлове сховище (Supabase Storage)

Оскільки бази даних (Postgres/Mongo) не призначені для зберігання бінарних файлів, весь медіаконтент платформи винесено
у спеціалізоване S3-сумісне сховище — **Supabase Storage**.

### 2.1. Структура та типи контенту

Усі файли логічно розділені по окремих "бакетах" (Buckets) залежно від сутності:

- `SUPABASE_USERS_PROFILE_PICTURES_BUCKET` — аватарки користувачів.
- `SUPABASE_COURSES_COVER_PICTURES_BUCKET` — обкладинки курсів та навчальні матеріали (відео, аудіо-лекції, презентації,
  PDF, DOC, TXT).

Для забезпечення чіткої організації простору та зручного управління контентом, файли зберігаються за багаторівневою
ієрархією. Шлях до файлу жорстко прив'язується до унікального ідентифікатора курсу (`course_id`), всередині якого
створюються логічні підпапки залежно від контексту використання (наприклад, `course_id/lesson/`,
`course_id/course-test/`, `course_id/module-test/`). Завдяки цьому всі матеріали курсу — від аудіолекцій до специфічних
зображень для тестових питань — логічно згруповані та ізольовані.

### 2.2. Стратегія доступу (Public URLs)

Всі завантажені навчальні файли та зображення мають публічний доступ. Це архітектурне рішення дозволяє:

- **Зняти навантаження з Django:** Фронтенд отримує пряме посилання (`public_url`) і завантажує важкі відео або картинки
  безпосередньо з серверів Supabase (CDN), взагалі не навантажуючи бекенд-сервер Smart-Study.
- **Спростити клієнтську логіку:** Зображення у тестах чи аватарки одразу готові до використання в тегах `<img>`.

### 2.3. Асинхронна інтеграція (Python Client)

Взаємодія бекенду зі сховищем реалізована не через стандартні синхронні бібліотеки, а за допомогою оптимізованих \*
\*асинхронних викликів\*\* (`sync_to_async`). Це критично важливо для підтримки високої пропускної здатності сервера під
час роботи з мережевими запитами.

Ключові реалізовані механізми:

1. **Завантаження (`handle_picture`):** Асинхронне зчитування файлу, видалення попередньої версії (якщо є) для економії
   місця, завантаження в бакет та збереження отриманого публічного URL у базу даних.
2. **Безпечне видалення (`delete_picture` & `get_all_objects_recursive`):** Оскільки S3-сховища не мають концепції "
   папок", реалізовано рекурсивний алгоритм пошуку. При видаленні курсу або профілю скрипт збирає всі файли за
   префіксом (`course_id/`) і видаляє їх пакетом, після чого видаляє "папки", запобігаючи накопиченню сміття (orphaned
   files).
3. **Пакетне переміщення (`move_supabase_files_batch`):** Дозволяє масово змінювати префікси файлів без необхідності
   завантажувати їх назад на бекенд-сервер, використовуючи внутрішній метод копіювання Supabase.

## 3. Результати впровадження

- **Масштабованість:** Архітектура повністю Stateless (без стану). Усі сесії, кеші та файли винесені в Redis та
  Supabase, що дозволяє запускати будь-яку кількість екземплярів Django-бекенду паралельно.
- **Швидкодія:** Віддача важкого медіаконтенту напряму з хмари та використання RAM-кешування для курсів зводить затримки
  для кінцевого користувача до мінімуму.
- **Асинхронність:** Перенесення розсилок електронної пошти та роботи з мережевим сховищем в асинхронний/фоновий режим
  звільнило ресурси сервера для миттєвої обробки клієнтських REST API запитів.
