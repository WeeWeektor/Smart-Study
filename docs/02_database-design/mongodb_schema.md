# Проєктування NoSQL бази даних (MongoDB)

У архітектурі **Smart-Study** база даних **MongoDB** використовується для зберігання навчального контенту, який має
гнучку, вкладену або ієрархічну структуру. На відміну від реляційної моделі, тут дані зберігаються у вигляді
JSON-подібних документів (BSON), що дозволяє завантажувати цілі блоки контенту (наприклад, весь тест або структуру
модуля) за один запит до бази.

## 1. Колекція `course_structures` (Структури курсів)

Ця колекція зберігає верхньорівневу ієрархію курсу. Оскільки курс може складатися як з модулів, так і з окремих тестів,
тут використовується патерн **поліморфного масиву** — поле `structure` містить об'єкти різних типів.

### Приклад документа:

```json
{
  "_id": {
    "$oid": "69e21d427f3bdc2b896b616e"
  },
  "structure": [
    {
      "type": "test",
      "test_id": "7d2e5060-3117-464f-9214-bed8c5926a75",
      "title": "Вступний тест",
      "order": 1,
      "time_limit": 3
    },
    {
      "type": "module",
      "module_id": "86010ae0-e0a8-4372-bebc-c7be3e7cf8ed",
      "title": "Основи програмування",
      "order": 2
    }
  ]
}
```

### Опис полів:

- **`_id`**: Унікальний ідентифікатор документа в MongoDB. Саме це значення (як рядок) зберігається в таблиці `courses`
  у PostgreSQL.
- **`structure`**: Масив елементів курсу.
- **`type`**: Дискримінатор типу (`test` або `module`), який вказує фронтенду, як саме рендерити цей елемент.
- **`test_id` / `module_id`**: UUID, які слугують посиланнями (Soft Links) на відповідні сутності в PostgreSQL для
  отримання їх метаданих.
- **`order`**: Порядковий номер елемента для правильного сортування в інтерфейсі (Drag-and-Drop).

---

## 2. Колекція `module_structures` (Структури модулів)

Ця колекція описує внутрішнє наповнення конкретного модуля. Вона дозволяє викладачам створювати довільну послідовність
уроків та проміжних тестів.

### Приклад документа:

```json
{
  "_id": {
    "$oid": "69e21d437f3bdc2b896b6170"
  },
  "structure": [
    {
      "type": "lesson",
      "lesson_id": "1dee4f94-4575-4387-a9d9-3938efa70643",
      "title": "Введення в тему",
      "order": 1,
      "content_type": "custom",
      "duration": "0:10:00"
    },
    {
      "type": "lesson",
      "lesson_id": "8fa7056e-6386-420c-902f-6bcbccb9b9eb",
      "title": "Аудіолекція",
      "order": 2,
      "content_type": "audio",
      "duration": "0:15:00"
    },
    {
      "type": "test",
      "test_id": "d5393c4e-f1c7-41b6-b2a9-26d51b88af30",
      "title": "Модульний контроль",
      "order": 3,
      "time_limit": 6
    }
  ]
}
```

### Опис полів:

- **`content_type`**: Додаткове поле для уроків, що визначає формат контенту (`custom`, `audio`, `presentation`, `document`,
  `video`, `image`). Це дозволяє клієнтському додатку обирати відповідний плеєр або компонент відображення.
- **`time_limit`**: Обмеження часу (у хвилинах) для виконання тесту, що зберігається безпосередньо в структурі для
  швидкого доступу без зайвих запитів.

## 3. Колекція `questions_data_for_test` (Дані тестів)

Найбільш динамічна частина бази даних. Збереження запитань у PostgreSQL вимагало б створення кількох пов'язаних
таблиць (Питання -> Варіанти відповідей -> Правильні відповіді). У MongoDB весь тест зберігається як єдиний,
самодостатній документ.

### Приклад документа:

```json
{
  "_id": {
    "$oid": "69e21d427f3bdc2b896b616f"
  },
  "questions": [
    {
      "order": 1,
      "type": "single",
      "questionText": "Чи підтримує Python множинне успадкування?",
      "image_url": "question_image_course_t1_q1",
      "choices": ["yes", "no"],
      "correctAnswers": ["no"],
      "points": 1,
      "explanation": ""
    },
    {
      "order": 2,
      "type": "multiple",
      "questionText": "Які з перелічених баз даних є NoSQL?",
      "image_url": "question_image_m2_t3_q1",
      "choices": ["PostgreSQL", "MongoDB", "Redis", "MySQL"],
      "correctAnswers": ["MongoDB", "Redis"],
      "points": 6,
      "explanation": "MongoDB (документо-орієнтована) та Redis (key-value)."
    }
  ]
}
```

### Опис полів:

- **`type`**: Тип запитання (`single` — одна правильна відповідь, `multiple` — декілька правильних відповідей).
- **`choices`**: Масив рядків, що містить усі можливі варіанти відповідей (використовується для рендеру
  чекбоксів/радіокнопок на фронтенді).
- **`correctAnswers`**: Масив правильних відповідей. На бекенді (Django) відбувається порівняння цього масиву з
  відповіддю студента для автоматичної перевірки.
- **`points`**: Вага запитання при розрахунку фінальної оцінки за тест.
- **`image_url`**: Опціональне поле з ключем для хмарного сховища (Supabase), якщо до запитання додається зображення.
- **`explanation`**: Додаткове поле для роз'яснення питання для студента (надання додаткової інформації або підказок).

## 4. Переваги такої структури

1. **Висока продуктивність:** При початку тестування бекенд дістає весь масив запитань (`questions`) за 1 операцію
   читання ($O(1)$) замість виконання важких `JOIN` запитів.
2. **Гнучкість (Schema-less):** Якщо в майбутньому з'явиться новий тип запитання (наприклад, "встановити відповідність"
   з об'єктами замість масивів), це не зламає поточну схему бази даних і не потребуватиме міграцій.
